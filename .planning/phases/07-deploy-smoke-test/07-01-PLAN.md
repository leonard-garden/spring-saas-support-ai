  <action>
    Append a SECOND entry to the `services:` list in `render.yaml`. Do not modify the existing backend entry.

    Insert exactly this block, indented to match the existing service (2-space list item):

    ```yaml
      - type: web
        name: support-saas-frontend
        runtime: static
        rootDir: frontend
        buildCommand: npm ci && npm run build
        staticPublishPath: dist
        pullRequestPreviewsEnabled: false
        routes:
          - type: rewrite
            source: /*
            destination: /index.html
        envVars:
          - key: VITE_API_URL
            sync: false
    ```

    Notes:
    - Use `runtime: static` (the modern Render Blueprint key). The legacy `env: static` from the research doc is deprecated.
    - `rootDir: frontend` is REQUIRED ÔÇö without it Render runs `npm ci` at repo root where there is no package.json for the frontend.
    - `staticPublishPath: dist` is relative to `rootDir`, so the resolved publish dir is `frontend/dist`.
    - The single `/*` Ôćĺ `/index.html` rewrite is the SPA fix. It must be the only route, or last.
    - `VITE_API_URL` is `sync: false` ÔÇö the operator sets it in the Render dashboard to the backend's public URL.

    Do NOT hardcode the backend URL inside render.yaml ÔÇö keep it operator-managed via `sync: false`.

    Commit message:
    ```
    chore(deploy): add frontend static-site service to render.yaml

    Adds support-saas-frontend service with SPA rewrite rule so client-side
    routes (/dashboard, /members, etc.) survive hard refresh on Render.

    Constraint: monorepo layout requires rootDir: frontend
    Constraint: Vite build emits to frontend/dist (default outDir)
    Rejected: hardcoding VITE_API_URL in yaml | makes blueprint env-coupled
    Confidence: high
    Scope-risk: narrow
    ```
